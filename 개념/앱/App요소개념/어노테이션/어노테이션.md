## 개념

* __어노테이션 = 코드에 붙이는 표시__

이 표시를 보고 컴파일러, Retrofit, Gson, Room, Hilt, Compose, Android Studio Lint*등이 특별한 처리를 한다.

- 붙는 위치

```kotlin
    // 클래스에 붙음
    @Entity
    data class User(...)

    // 함수/메소드에 붙음
    @GET("users")
    suspend fun getUsers(): List<UserDto>

    // 파라미터에 붙음
    fun saveUser(@Body request: UserRequestDto)

    // 변수/필드에 붙음
    @SerializedName("user_name")
    val userName: String
```

-> 어노테이션은 ___클래스, 함수, 변수, 파라미터___ 등에 붙일 수 있다.

# 자주 쓰는 어노테이션

### @SerialzedName

- ___데이터를 보낼 때 (직렬화)와 받을 때(역직렬화)___ 모두 작동하며, 양방향 모두에서 이름을 ___매핑(연결)___ 해 주는 기능

- Java/Kotlin의 변수명과 서버 JSON의 키(Key) 이름을 __매핑__ 할 때 씀

- 예시

```kotlin
import com.google.gson.annotations.SerializedName

data class GitHubProfileRequestDto(

    // JSON key 이름을 "github_Id"로 지정한다.
    // 서버에 보낼 때 { "github_Id": "cnsvkf" } 형태가 된다.
    @SerializedName("github_Id")
    val githubId: String
)
```

: __Kotlin에서는 편하게 githubId로 쓰고__, 서버와 통신할 때는 "github_id"로 변환된다.

\+

서버가 snake_case를 쓴다고 하자.( _ 쓰기)

    {
    "github_id": "cnsvkf"
    }
Kotlin에서는 보통 camelCase를 쓴다. (githubId)

### @Expose

- __Gson에서 직렬화/역직렬화 대상__ 인지 표시할 때 쓴다.
    - 직렬화: 객체(Object) ➡️ 문자열/바이트 (보내기 좋은 형태로 변환)

    - 역직렬화: 문자열/바이트 ➡️ 객체(Object) (쓰기 좋은 형태로 변환)

```kotlin
import com.google.gson.annotations.Expose
import com.google.gson.annotations.SerializedName

data class UserProfile(
    // 1. 모든 통신에서 정상적으로 포함시킬 필드들
    @SerializedName("id")
    @Expose
    val id: Int,

    @SerializedName("email")
    @Expose
    val email: String,

    // 2. 서버에서 다운로드(역직렬화)는 받지만, 서버로 업로드(직렬화)는 막을 필드
    @SerializedName("password")
    @Expose(deserialize = true, serialize = false) 
    val password: String,

    // 3. 앱 내부 로직용 변수 (서버와 주고받는 모든 과정에서 제외)
    // @Expose를 붙이지 않으면 자동으로 제외됩니다.
    val isLocalChecked: Boolean = false
)
)
```

* 중요

    @Expose는 Gson 설정에서 .___excludeFieldsWithoutExposeAnnotation()___ 을 사용해야 의미가 커진다. (__빌더설정 켜야함__) 

\+ @SerialedName 을 더 자주 씀 | 데이터를 세밀하게 설정할 떄 사용

### @Query

- __URL 뒤에 ?key=value 형태__ 로 값을 붙일 때 사용한다.

-> 쓰는 이유

    직접 URL을 만들면 위험함

    나쁜 예시:
    val url = "https://api.example.com/users?name=$name&age=$age"

    val name = "park jisung"
    공백이 있으면 URL에서는 그대로 쓰면 안됨.

    https://api.example.com/users?name=park jisung
    URL에는 공백이 들어가면 안 되기 때문에 인코딩이 필요하다.

    정상 URL:
    https://api.example.com/users?name=park%20jisung
    @Query를 쓰면 Retrofit이 이런 URL 인코딩을 자동으로 처리해준다.

-> 예시

GET https://api.example.com/users?name=park&age=18

```kotlin
// 메소드 호출
@GET("users")
suspend fun getUsers(
    @Query("name") name: String,
    @Query("age") age: Int
): Response<List<UserDto>>

// 입력 예시
apiService.getUsers(
    name = "park",
    age = 18
)
```

-> https://api.example.com/users?name=park&age=18 : 실제 요청 URL

| 구분       | 사용 위치      | 예시                 |
| -------- | ---------- | ------------------ |
| `@Path`  | URL 경로 안   | `/users/15`        |
| `@Query` | URL 뒤 조건 값 | `/users?name=park` |

    ->
    @Path  = 어떤 자원을 볼지 정함
    @Query = 그 자원을 어떤 조건으로 필터링할지 정함

-> __예시__

```kotlin
@GET("users/{userId}/posts")
suspend fun getUserPosts(
    @Path("userId") userId: Long,
    @Query("page") page: Int,
    @Query("size") size: Int
): Response<List<PostDto>>
```

: userId = 누구의 게시글인지
page, size = 몇 번째 페이지를 몇 개 가져올지

### @Headers / @Header

1.  @Header : 요청마다 헤더 값을 직접 넣을 때 사용한다.
    - 실무에서는 매번 @Header로 넣기보다 OkHttp Interceptor에서 자동으로 토큰을 붙이는 경우가 많다.

-> 예시 

```kotlin
@GET("users/me")
suspend fun getMyProfile(
    // Authorization 헤더에 accessToken을 넣는다.
    @Header("Authorization") accessToken: String
): Response<UserDto>
```
호출 :

```kotlin
api.getMyProfile("Bearer abc.def.ghi")
```
2. @Headers : 고정 헤더를 넣을 때 사용

-> 예시

```kotlin
@Headers("Content-Type: application/json")
@POST("auth/login")
suspend fun login(
    @Body request: LoginRequestDto
): Response<LoginResponseDto>
```

### @FormUrlEcoded / @Field

- 폼 데이터 형식으로 보낼 떄 사용한다.

1. @FormUrlEncoded : ___JSON이 아니라 Form 방식으로 Body___ 를 보낸다고 Retrofit에게 알려주는 어노테이션

2. @Field : Form 방식 요청에서 각각의 값을 ___key-value 형태___ 로 보내는 어노테이션

```kotlin
package com.example.app.data.remote

import com.example.app.data.model.LoginResponseDto
import retrofit2.Response
import retrofit2.http.Field
import retrofit2.http.FormUrlEncoded
import retrofit2.http.POST

interface AuthApiService {

    // 이 요청의 Body를 application/x-www-form-urlencoded 형식으로 보낸다는 뜻
    @FormUrlEncoded

    // POST 방식으로 /auth/login API에 요청을 보냄
    @POST("auth/login")
    suspend fun loginByForm(
        // 서버에 id=입력값 형태로 전송됨
        @Field("id") id: String,

        // 서버에 password=입력값 형태로 전송됨
        @Field("password") password: String
    ): Response<LoginResponseDto>
}
```

서버로 보내지는 형태 :

    POST /auth/login
    Content-Type: application/x-www-form-urlencoded

    id=cnsvkf&password=1234

요즘 앱에서는 JSON body 방식인 @Body를 더 많이 쓴다.

| 어노테이션             | 보내는 방식                   | Content-Type                        | 주로 쓰는 상황                            |
| ----------------- | ------------------------ | ----------------------------------- | ----------------------------------- |
| `@Body`           | 객체를 통째로 Request Body에 넣음 | 보통 `application/json`               | 로그인, 회원가입, 게시글 작성처럼 JSON으로 서버에 보낼 때 |
| `@FormUrlEncoded` | Form 방식으로 보낸다고 메소드에 표시   | `application/x-www-form-urlencoded` | 서버가 form-data key-value 형식을 요구할 때   |
| `@Field`          | Form 안의 개별 key-value 값   | `application/x-www-form-urlencoded` | `id=abc&password=1234`처럼 필드별로 보낼 때  |

Form 방식은 데이터를 서버에 보낼 때 객체 JSON으로 보내는 게 아니라, key=value 형태로 나눠서 보내는 방식

### @Multipart / @Part

- 이미지나 파일 업로드할 때 사용한다.


-> 파일 업로드 예시

```kotlin
@Multipart
@POST("users/profile/image")
suspend fun uploadProfileImage(

    // 이미지 파일 데이터를 multipart part로 보낸다.
    @Part image: MultipartBody.Part
): Response<Unit>
```

->

```kotlin
// 이미지 파일을 서버에 업로드하기 위한 RequestBody를 만든다.
val requestBody = file.asRequestBody("image/*".toMediaTypeOrNull())

// multipart/form-data 형식의 파일 part를 만든다.
val imagePart = MultipartBody.Part.createFormData(
    name = "image",
    filename = file.name,
    body = requestBody
)

// 서버에 이미지 업로드 요청을 보낸다.
api.uploadProfileImage(imagePart)
```

| 어노테이션                | 역할                                     |
| -------------------- | -------------------------------------- |
| `@Multipart`         | 이 API 요청이 multipart/form-data 방식이라는 표시 |
| `@Part`              | multipart 요청 안에 들어가는 각각의 데이터 조각        |
| `@Part("name")`      | 텍스트 값이나 단순 데이터를 보낼 때 사용                |
| `MultipartBody.Part` | 이미지, PDF, 파일 같은 실제 파일 조각               |
| `RequestBody`        | 텍스트나 파일을 HTTP 요청 바디 형태로 변환한 객체         |

    POST /users/profile
    Content-Type: multipart/form-data

    Part 1: nickname = "박지성"
    Part 2: githubId = "cnsvkf"
    Part 3: profileImage = profile.png 파일

즉, @Part 하나가 데이터 조각 하나

| 상황                    | 추천 방식                        |
| --------------------- | ---------------------------- |
| 로그인, 회원가입, 게시글 작성     | `@Body`                      |
| 간단한 key-value form 전송 | `@FormUrlEncoded` + `@Field` |
| 이미지, 파일 업로드           | `@Multipart` + `@Part`       |
| 이미지 + 텍스트 같이 업로드      | `@Multipart` + `@Part`       |

### @OptIn

- 아직 ___실험적이거나 안정화되지 않은 API를 사용___ 할 때 붙인다.

    - 예를 들어 __Material3의 일부 API, Foundation의 일부 API를 사용__ 할 때 필요할 수 있다.

        - ExperimentalMaterial3Api는 Material3에서 제공하는 실험적 API 표시용 어노테이션이다. Android 공식 문서에도 “이 Material API는 실험적이며 미래에 변경되거나 제거될 수 있다”는 메시지로 @RequiresOptIn이 붙어 있다.


-> 이걸 붙이면 컴파일러 경고/오류가 사라진다.

\+ 

여기서 API는 서버 API가 아니라, Compose Material3 라이브러리에서 제공하는 함수,*클래스, 어노테이션, 파라미터 같은 것을 말한다.

```kotlin
@OptIn(ExperimentalMaterial3Api::class)
@Composable
fun ExampleTopAppBar() {
    CenterAlignedTopAppBar(
        title = {
            Text(text = "GitHub Profile")
        }
    )
}
```

### @Stable 

- Compose에게 이 타입은 __안정적인 상태__ 라고 알려줄 때 사용한다.
    
    - Compose 성능 최적화 단계

```kotlin
@Stable
data class GitHubProfileUiState(
    val githubId: String = "",
    val isLoading: Boolean = false
)
```
->

값이 안 바뀌었으면 이 Composable을 다시 실행하지 않아도 된다”라고 Compose에게 알려주기 위해서.

-> ___조건___(중요)

    공식 API 기준으로 @Stable 클래스나 인터페이스는 아래 조건을 만족해야 한다.

| 공식 조건                             | 쉬운 말                                             |
| --------------------------------- | ------------------------------------------------ |
| `equals()` 결과가 일관적이어야 함           | 같은 값이면 항상 같다고 나와야 함                              |
| public 프로퍼티 값이 바뀌면 Compose가 알아야 함 | 일반 `var` 말고 `mutableStateOf` 같은 Compose 상태를 써야 함 |
| public 프로퍼티 타입들도 stable이어야 함      | 안에 들어있는 객체도 안전한 구조여야 함                           |

-> 

    @Immutable
    → 값이 안 바뀌는 객체

    @Stable
    → 값이 바뀔 수 있지만, 바뀐 사실을 Compose가 알 수 있는 객체
    : 따라서 2번 조건이 성립함
@Stable은 값이 바뀔 수는 있지만, 바뀌면 Compose가 알 수 있다는 의미에 가깝다.

\+ @Stable, @Immutable은 컴파일러와의 약속이며, 잘못 붙이면 예상한 재구성이 일어나지 않을 수 있다고 경고한다

### @Immutable

- 값이 변하지 않는 타입이라는 것을 Compose에게 알려준다.

-> 예시(UI)

```kotlin
import androidx.compose.runtime.Immutable

@Immutable
data class LoginUiState(
    val id: String = "",
    val password: String = "",
    val isLoading: Boolean = false,
    val errorMessage: String? = null
)
```
```kotlin
@Composable
fun LoginScreen(
    uiState: LoginUiState,
    onIdChange: (String) -> Unit,
    onPasswordChange: (String) -> Unit,
    onLoginClick: () -> Unit
) {
    Column {
        TextField(
            value = uiState.id,
            onValueChange = onIdChange
        )

        TextField(
            value = uiState.password,
            onValueChange = onPasswordChange
        )

        Button(
            onClick = onLoginClick,
            enabled = !uiState.isLoading
        ) {
            Text("로그인")
        }

        uiState.errorMessage?.let {
            Text(text = it)
        }
    }
}
```

비밀번호 값은 바뀌지만, 기존 LoginUiState 객체 안의 password 값을 직접 수정하는 게 아니라
새로운 LoginUiState 객체를 만들어 교체한다.
